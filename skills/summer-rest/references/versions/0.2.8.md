# Summer 0.2.8 — summer-rest

Released: 2026-04-12

## Features

### `SummerGlobalExceptionHandler` — richer `ServerWebInputException` handling

`ServerWebInputException` now extracts field-level error details from the Jackson cause chain:

| Cause | Extracted |
|---|---|
| `InvalidFormatException` | field path, rejected value, target type |
| `JsonMappingException` (other) | original message |
| `JsonParseException` | parse location |

Previously these landed in the response as opaque `400 Bad Request` with the bare exception
message. Now consumers see structured `details` in the response body.

Sample response for a malformed numeric field:

```json
{
  "code": "validation.error",
  "message": "Invalid request body",
  "timestamp": "2026-04-12T10:00:00Z",
  "details": [
    {"field": "amount", "issue": "Cannot deserialize value 'abc' as java.math.BigDecimal"}
  ]
}
```

## Breaking changes

None for `summer-rest`. Related cross-cutting changes:

- `Ufid.fromString()` no longer detects 32-char hex format. Endpoints binding `Ufid` from
  inputs that were hex-encoded must switch callers to UUID or Base32 before 0.2.8.
- `UfidFormat` → `UfidDisplay` rename — not visible to controllers, only affects code that
  imported the formatter directly.

See `summer-core/references/versions/0.2.8.md` for the full Ufid changes.

## Migration from 0.2.6

No required changes for REST controllers. To take advantage of the richer error details, ensure
your global error response DTO surfaces the `details` field (already the case if you use
`JsonErrorResponse` directly from `core.exception`).
