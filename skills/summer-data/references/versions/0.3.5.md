# Summer 0.3.5 — summer-data (R2DBC `Txid` converters)

Released: 2026-05-10

R2DBC converters for the new `Txid` type land in `summer-data-r2dbc`. Both readers always register; the writer side is exclusive and gated by a typed property.

## Features

### `TxidConverter` (`Long ↔ Txid`)

For `BIGINT` columns. **Preferred for new schemas** — 4× smaller index entries vs UUID, native int comparison.

### `TxidUuidConverter` (`UUID ↔ Txid`)

For legacy `UUID`-typed columns. Packs the 59-bit `Txid` into a UUID with `MSB = 0` (matches `Txid.toUUID()` exactly).

### `SummerR2dbcProperties` typed config

```yaml
summer:
  r2dbc:
    txid-column-type: uuid     # default
    # txid-column-type: bigint
```

`@ConfigurationProperties("summer.r2dbc")` bean exposing `TxidColumnType` enum (`UUID` or `BIGINT`, case-insensitive binding). Any other value fails at context start with a `BindException` that names both the property and the offending value.

## Behavior

**Reads are always registered.** The R2DBC driver yields `Long` for BIGINT and `UUID` for UUID, so the matching `Reading` converter is selected by source type — no ambiguity.

**Writes are exclusive.** Spring Data R2DBC's `MappingR2dbcConverter` resolves writers by the property's Java type only (no SQL-type hint), so registering both writers leaves the choice to registration order and silently binds the wrong SQL type. To eliminate that, exactly **one** writer is registered:

| `txid-column-type` | Writer registered |
|---|---|
| `uuid` (default) | `TxidUuidConverter.Writing` |
| `bigint` | `TxidConverter.Writing` |
| other | startup fails with `BindException` |

Mixing BIGINT and UUID `Txid` columns in the **same app** is supported on the read side but not on the write side — standardize on one per app.

**Default is `uuid`** because the existing eWallet PK convention is UUID-typed (saga/account/hold/va ids all UUID); set `bigint` on greenfield schemas that want the smaller index.

## Source layout

- `io.f8a.summer.data.r2dbc.config.SummerR2dbcProperties`
- `io.f8a.summer.data.r2dbc.config.TxidColumnType`
- `SummerR2dbcAutoConfiguration` — `@EnableConfigurationProperties(SummerR2dbcProperties.class)` and switches on `properties.getTxidColumnType()` to pick the writer.

## Breaking changes

None — purely additive within `summer-data`. The breaking change is on the payment-sdk side (`LedgerTransferEvent.Transfer.{id, pendingId}` type), see `summer-payment-sdk/references/versions/0.3.4.md`.

## Migration from 0.3.4

1. **Decide column type once per app.** Greenfield → `bigint`. Legacy UUID-PK schemas → leave the default `uuid`.

   ```yaml
   summer:
     r2dbc:
       txid-column-type: bigint   # greenfield
   ```

2. **DDL for new tables:**

   ```sql
   CREATE TABLE transactions (
       txid BIGINT NOT NULL PRIMARY KEY,
       ufid UUID   NOT NULL UNIQUE,           -- canonical internal key (Ufid)
       amount NUMERIC(20,8) NOT NULL,
       created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
   );
   ```

   For legacy schemas standardized on UUID:

   ```sql
   CREATE TABLE transactions (
       txid UUID NOT NULL PRIMARY KEY,        -- TxidUuidConverter packs Txid into UUID, MSB=0
       ...
   );
   ```

3. **Entity:**

   ```java
   @Table("transactions")
   public record Transaction(
       @Id Txid txid,
       Ufid customerId,
       BigDecimal amount,
       Instant createdAt) {}
   ```

4. **Wire `TxidGenerator`** (in `summer-core/references/versions/0.3.5.md`) and mint via `txidGen.next()`.

5. **Verify the writer at boot** — set log level `io.f8a.summer.data.r2dbc=DEBUG`; you should see one of `TxidConverter.Writing registered` or `TxidUuidConverter.Writing registered`.

## Verification

- A `Txid`-typed `@Id` round-trips through R2DBC against both `BIGINT` and `UUID` columns, depending on the property.
- Setting `summer.r2dbc.txid-column-type=bogus` fails boot with `BindException` naming the property and the value.
- A schema with **both** a `BIGINT` Txid column and a `UUID` Txid column reads correctly from both, but writes fail for whichever doesn't match the property — surface this as "pick one or split the apps."
