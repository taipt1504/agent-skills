# Summer 0.2.8 — summer-data

Released: 2026-04-12

This release introduces **Debezium CDC** as an alternative outbox publishing mode and overhauls
`OutboxProperties`. The schema config in this version is replaced again in 0.3.1 — see that
overlay for the current canon.

## Features

### Debezium CDC outbox mode

```yaml
f8a.outbox.publisher.mode: cdc       # default: scheduler
```

Streams events directly from the PostgreSQL WAL instead of polling. Requires
`wal_level=logical` and Debezium dependencies on the classpath. Only one CDC instance per
replication slot; use `scheduler` mode for multi-instance deployments.

`OutboxScheduledTasks` split into:
- `OutboxPublisherTask` — scheduler-mode initial publishes only (`retry_count=0`).
- `OutboxMaintenanceTasks` — runs in **both** modes (cleanup, monitoring).

CDC mode cleanup: `deleteOldEvents` (any age). Scheduler mode: `deletePublishedEvents`.

### Unified retry + backoff (works in both modes)

New `OutboxRetryTask` runs in both modes and picks up events that previously failed
(`published=false AND retry_count BETWEEN 1 AND max-1 AND next_retry_at <= now()`), applying
exponential backoff via `OutboxBackoff` (same formula as Spring Kafka's
`ExponentialBackOffWithMaxRetries`, zero new dependency).

Closes the CDC-mode gap where publish failures were silently dropped — `markAsFailed` now
schedules `next_retry_at` and the retry task recovers the event.

`OutboxPublisherTask` narrowed to initial publishes only. Retries no longer loop on every 5s
poll — they honour the unified backoff policy.

### Outbox event id changed UUID → Ufid

`OutboxEvent.id` is now a `Ufid` generated via `Ufid.generate()`. `markAsPublished()` and
`markAsFailed()` accept `Ufid` instead of `UUID`. The database column remains `UUID` (Ufid
stores as UUID).

### New OutboxService methods

- `findNewEvents(int)` — fetch un-published events up to `limit`.
- `findEventsDueForRetry(int, int)` — fetch retry-eligible events.
- `markAsFailed(Ufid, String, Instant)` overload — accepts an explicit `next_retry_at`. The
  legacy two-arg form still works (treats `null` as "retry immediately").

### Schema additions

```sql
ALTER TABLE outbox_events ADD COLUMN IF NOT EXISTS next_retry_at TIMESTAMPTZ;
CREATE INDEX IF NOT EXISTS idx_outbox_retry
    ON outbox_events (next_retry_at)
    WHERE published = false AND retry_count > 0;
```

### New retry config block

```yaml
f8a.outbox.retry:
  max-attempts: 5
  initial-interval: 1s          # was *-ms long pre-0.2.8
  multiplier: 2.0
  max-interval: 60s
  poll-interval: 10s            # retry-task cadence
  batch-size: 100
```

Replaces the implicit "retry forever every 5s" scheduler-mode behaviour with a bounded,
backoff-aware policy.

## Breaking changes

### Removed `f8a.outbox.publisher.enabled`

To disable publishing now, set `f8a.outbox.enabled: false` (disables the entire outbox module)
or simply don't provide an `OutboxEventPublisher` bean (publisher + retry tasks won't be
created).

### `OutboxProperties` redesign

The `scheduler.*` umbrella is gone (it was misleading — `cleanup` and `failed-events` actually
ran in both modes). Properties grouped by concern, all time values use `Duration` (suffixed
strings like `30d`, `10s`) instead of mixed `*-ms` / `*-seconds` / `*-days` fields.

| Old | New |
|---|---|
| `f8a.outbox.scheduler.publisher.cron` | `f8a.outbox.publisher.cron` |
| `f8a.outbox.scheduler.publisher.initial-delay` | `f8a.outbox.publisher.initial-delay` |
| `f8a.outbox.scheduler.cleanup.cron` | `f8a.outbox.cleanup.cron` |
| `f8a.outbox.scheduler.cleanup.retention-days: 30` | `f8a.outbox.cleanup.retention: 30d` |
| `f8a.outbox.scheduler.failed-events.cron` | `f8a.outbox.monitoring.cron` |
| `f8a.outbox.scheduler.failed-events.max-retry-threshold` | *removed* — use `f8a.outbox.retry.max-attempts` |
| `f8a.outbox.retry.initial-interval-ms: 1000` | `f8a.outbox.retry.initial-interval: 1s` |
| `f8a.outbox.retry.max-interval-ms: 60000` | `f8a.outbox.retry.max-interval: 60s` |
| `f8a.outbox.retry.interval-ms: 10000` | `f8a.outbox.retry.poll-interval: 10s` |
| `f8a.outbox.circuit-breaker.wait-duration-seconds: 60` | `f8a.outbox.circuit-breaker.wait-duration: 60s` |

### Outbox publisher uses `concatMap` (was `flatMap`)

Events publish sequentially in `created_at` order, using one R2DBC connection at a time. Avoids
the connection-fan-out behaviour of `flatMap` under bursty loads.

## Migration from 0.2.6

1. Apply schema migration (`next_retry_at` + retry index).
2. Rewrite `f8a.outbox.scheduler.*` and `f8a.outbox.retry.*-ms`/`*-seconds`/`*-days` keys per
   the table above.
3. Delete `f8a.outbox.publisher.enabled` from configs — set `f8a.outbox.enabled: false` if you
   want the module disabled.
4. Direct callers of `markAsPublished(UUID)` / `markAsFailed(UUID, ...)`: switch the `id` arg
   type to `Ufid`.
5. (Optional) opt into CDC by setting `f8a.outbox.publisher.mode: cdc` and providing Debezium
   deps + `wal_level=logical`.
