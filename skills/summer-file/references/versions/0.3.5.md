# Summer 0.3.5 — summer-file (new module)

Released: 2026-05-10

`summer-file` ships in 0.3.5. There is no prior version to migrate from — the module is purely additive.

## Features

### Streaming zip exporter (`io.f8a.summer.file.export`)

Generic size-capped, chunked archive writer for service exports. Public API:

- `ExportRow` (marker)
- `DatedExportRow extends ExportRow` (sub-interface with `LocalDate date()`)
- `ChunkWriter<R>` (functional — caller owns per-file format)
- `ExportSpec<R>` (`@Builder`: `baseName`, `fileExtension`, `maxRowsPerFile=100_000`, `maxZipSizeBytes`, `chunkWriter`)
- `ZipExporter` — static `writeZip(...)` and `pipeZip(...)`; both `Iterable<R>` and `Flux<R>` overloads
- `SizeLimitedOutputStream` — `FilterOutputStream`, aborts past `maxBytes`

Handles date grouping, per-file row caps, total zip-size caps, and the pipe-to-uploader pattern via `PipedInputStream`. Empty input still emits one entry so the file always carries the header row.

The `Flux<R>` overload bridges via `Flux.toIterable()` and **must** run on a blocking-capable scheduler (`Schedulers.boundedElastic()`).

## Breaking changes

None. New module.

## Migration from 0.3.4

Optional adoption — to replace service-local zip plumbing:

1. Add the dependency: `implementation 'io.f8a.summer:summer-file'`.
2. Locate the existing `ZipOutputStream` setup (most exporting services have one) and replace with `ZipExporter.writeZip(...)` (sync) or `ZipExporter.pipeZip(...)` (async upload).
3. Wrap your per-format writer (POI xlsx, OpenCSV csv, …) in a `ChunkWriter<R>` lambda. **Do not close the `OutputStream`** — the exporter owns it.
4. Pick caps deliberately:
   - `maxRowsPerFile` ≤ ~50–100k for Excel compatibility.
   - `maxZipSizeBytes` ≤ ~25 MB if the file rides email; otherwise size against your object-storage / DLP limits.

No DB schema changes.

## Verification

- A run over an empty `Iterable`/`Flux` still produces a zip with one entry containing only the header.
- `maxZipSizeBytes` is honoured — an oversize run fails with `IOException("Size limit exceeded ...")` mid-stream.
- For `pipeZip`, killing the writer side propagates `"Pipe closed"` to the uploader.
