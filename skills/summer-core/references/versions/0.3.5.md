# Summer 0.3.5 — summer-core (`Txid` family + `summer-file` module)

Released: 2026-05-10

Two unrelated, generally-useful primitives bundled in one release:
1. **`Txid`** — Snowflake-style 59-bit human-facing transaction reference, distinct from `Ufid`.
2. **`summer-file`** — new module with a streaming zip exporter (covered in `summer-file/references/versions/0.3.5.md`).

This overlay covers the summer-core surface. R2DBC converter wiring lives in `summer-data/references/versions/0.3.5.md`.

## Features

### `Txid` — 59-bit transaction reference

Package: `io.f8a.summer.core.domain.types.txid`. Final value class, `Comparable<Txid>`, `Serializable`. `@JsonValue` / `@JsonCreator` produce/parse the 18-digit zero-padded decimal.

**Bit layout (59 bits — fits 18 decimal digits):**
```
[58:18] 41-bit timestamp ms (since EPOCH_MS = 2026-01-01T00:00:00Z)
[17:12]  6-bit machine ID  (0..63)
[11:0]  12-bit per-ms seq   (0..4095)
```

Throughput: 4,096 IDs/ms/machine × 64 machines = 262,144 IDs/ms cluster-wide (~262M/sec). The 18 bits below the timestamp are fixed at 262,144 values total; `MACHINE_BITS` vs `SEQUENCE_BITS` is a *fleet-size vs per-pod-throughput* trade-off, not a capacity one.

**Encodings:**

| Method | Direction | Storage |
|---|---|---|
| `of(long)` / `toLong()` | ↔ | `BIGINT` |
| `fromString(String)` / `toString()` / `toJsonValue()` | ↔ | display (18-digit zero-padded decimal) |
| `fromUUID(UUID)` / `toUUID()` | ↔ | `UUID` (MSB = 0) |
| `from16Bytes(byte[])` / `as16Bytes()` | ↔ | **TigerBeetle u128 little-endian wire format**, high 8 bytes always zero |
| `fromUInt128(BigInteger)` / `toUInt128()` | ↔ | unsigned 128-bit `BigInteger` |
| `getInstant()`, `getMachineId()`, `getSequence()` | decode | debugging |

All `from*` constructors throw on a non-zero high half — a `Txid` produced by `TxidGenerator` always has the upper 64 bits zero, so a non-zero half means the value came from somewhere else (legacy, manual insert, foreign service) and the bug should surface immediately.

### `TxidGenerator`

Synchronized monotonic generator. **One per JVM.** Two constructors:

- `TxidGenerator(long machineId)` — fixed identity. Use with `MachineIdResolver.resolve()` on StatefulSets.
- `TxidGenerator(Supplier<Long> machineIdSupplier)` — dynamic identity, re-evaluated per `nextLong()` call. Use with `RedisMachineIdReservation` on autoscaled fleets.

Throughput ceiling 4096 IDs/ms/machine via busy-wait on `Thread.onSpinWait()`. Clock-backwards by ≤ 100 ms is tolerated (busy-wait to catch up); larger skew throws `IllegalStateException` rather than risk duplicates.

**Sequence storage is in-memory only** (`private long sequence` on the instance). Nothing on disk, nothing in Redis. Uniqueness is guaranteed by the `(machineId, timestamp_ms, sequence)` tuple — the wall clock acts as the persistent counter. Across restarts, time advances → new milliseconds → fresh sequence space.

### `MachineIdResolver`

Static machineId resolver. Sources in priority order:

1. `SUMMER_TXID_MACHINE_ID` env var.
2. `summer.txid.machine-id` system property.
3. Kubernetes StatefulSet ordinal — parses `HOSTNAME=<svc>-<n>` (e.g. `wallet-0` → 0).

Throws `IllegalStateException` if none of those is set. **Never silently hashes the hostname** — two pods with the same machineId is silent ID corruption, so the resolver fails loud instead of inventing one.

### `RedisMachineIdReservation`

Self-healing distributed lease (`Closeable + Supplier<Long>`). Scans `<prefix>:0..63` with `SET NX EX <ttl>`, claims the first free slot, refreshes every `ttl/3` (Redisson / Curator / etcd ratio). If a refresh observes the key is gone (TTL expired during GC pause / network partition / Redis restart), the lease **re-acquires a fresh slot atomically** — `TxidGenerator` (via the `Supplier<Long>` constructor) picks up the new id on its next `next()` call. No pod restart required.

Pass directly to `TxidGenerator(Supplier)`. Implements `Closeable` (graceful release deletes the key) and `Supplier<Long>` (passes directly to the generator). `Supplier.get()` throws if `isHealthy()` flips false (no successful refresh within `ttl`), so a generator using the supplier form fails loud rather than colliding with another pod that already reclaimed the slot.

### Payment-SDK Transfer.id is `Txid` (BREAKING, paired with payment-sdk 0.3.4)

`LedgerTransferEvent.Transfer.id` / `pendingId` are now `Txid` (were `Ufid`). Wire JSON is the 18-digit zero-padded decimal from `Txid.toJsonValue()`, not a UUID. Every consumer of `ledger.transfer.posted` (and siblings) must be on payment-sdk `0.3.4`+.

`Account.id` stays `Ufid` — TigerBeetle accounts can predate the Txid cutoff and aren't all generated by `TxidGenerator`.

## Wiring patterns

### A. Static (StatefulSet / configmap-driven Deployment)

```java
@Configuration
public class IdConfiguration {
  @Bean
  public TxidGenerator txidGenerator() {
    long machineId = MachineIdResolver.resolve();
    log.info("TxidGenerator initialized with machineId={}", machineId);
    return new TxidGenerator(machineId);
  }
}
```

### B. Dynamic (vanilla Deployment / Cloud Run / autoscaled)

```java
@Configuration
public class IdConfiguration {
  @Bean(destroyMethod = "close")
  public RedisMachineIdReservation txidLease(ReactiveStringRedisTemplate redis) {
    return new RedisMachineIdReservation(
        redis,
        "transfer-id-generator",         // key prefix per fleet
        Duration.ofSeconds(30));         // ttl; refresh interval auto-derived as ttl/3
  }

  @Bean
  public TxidGenerator txidGenerator(RedisMachineIdReservation lease) {
    return new TxidGenerator(lease);
  }
}
```

Use `RedisMachineIdReservation.isHealthy()` as a Kubernetes liveness probe.

## Migration from 0.3.4 (adopting `Txid`)

1. Pick a column type. **Greenfield:** `BIGINT NOT NULL PRIMARY KEY` (4× smaller index, native int comparison). **Legacy schemas standardized on UUID:** keep `UUID` and rely on `TxidUuidConverter`.
   ```sql
   CREATE TABLE transactions (
       txid BIGINT NOT NULL PRIMARY KEY,
       ufid UUID   NOT NULL UNIQUE,        -- canonical internal key (Ufid)
       ...
   );
   ```
2. Wire `TxidGenerator` (snippet above). One-time setup.
3. Configure machineId source: **StatefulSet** — nothing to do; **Deployment** — inject `SUMMER_TXID_MACHINE_ID` per pod, or use `RedisMachineIdReservation`.
4. Use `Txid` in entities — R2DBC converters are auto-registered. See `summer-data/references/versions/0.3.5.md` for the `summer.r2dbc.txid-column-type` property.
5. **Don't change `EPOCH_MS` after first deploy.** The custom epoch is a permanent decision; shifting it later retroactively renames every existing ID's timestamp.
6. **Don't reuse retired pod ordinals immediately** on scale-down. Wait at least 100 ms before recycling — otherwise the new pod could mint IDs that collide with the old pod's tail. (StatefulSets handle this naturally because the rolling restart waits for pods to terminate.)
7. (Optional) adopt `summer-file` to replace service-local `ZipOutputStream` plumbing. See `summer-file/references/versions/0.3.5.md`.

## TigerBeetle interop

`as16Bytes()` / `from16Bytes(byte[])` produce/consume the **little-endian** 16-byte layout TigerBeetle uses for `u128` ids. A `Txid` minted by `TxidGenerator` is therefore a valid TigerBeetle transfer/account id — the upper 64 bits are always zero. Mint a `Txid` in your service, hand it to TigerBeetle as the transfer's `id`, and later read the same value from the TB CDC stream:

```java
Txid txid = txidGen.next();
byte[] tbId = txid.as16Bytes();           // wire format for TB

// Reading back from CDC:
Txid sameTxid = Txid.from16Bytes(tbId);   // throws if upper 8 bytes != 0 (foreign id)
```

## Verification

- `MachineIdResolver.resolve()` in a StatefulSet pod with `HOSTNAME=wallet-0` returns `0`.
- `TxidGenerator.next().getInstant()` equals "now" within a millisecond.
- `Txid.fromString("000123456789012345")` round-trips through `toString()`.
- `from16Bytes(...)` throws on a TB transfer id minted before the cutoff (upper 8 bytes non-zero).
- `RedisMachineIdReservation` re-acquires after deleting its slot key in Redis (`redis-cli DEL transfer-id-generator:<n>`) — the next `txidGen.next()` carries a different `getMachineId()`.
