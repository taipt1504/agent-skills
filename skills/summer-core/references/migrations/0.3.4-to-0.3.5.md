# Migration: Summer 0.3.4 â†’ 0.3.5

Two adoptions, both opt-in: `Txid` family and `summer-file`. Plus one **mandatory** payment-sdk bump for any service that consumes `ledger.transfer.*` events.

## 1. Bump the BOM

```gradle
implementation platform('io.f8a.summer:summer-platform:0.3.5')
```

`paymentSdkVersion` is pinned to `0.3.4` by this BOM and brings the `Transfer.id` / `pendingId` type change (`Ufid` â†’ `Txid`).

## 2. Mandatory if you consume `ledger.transfer.*` events: walk Transfer.id type errors

The wire JSON of `LedgerTransferEvent.Transfer.id` / `pendingId` changes from a UUID string to the 18-digit decimal from `Txid.toJsonValue()`. **Every consumer** of `ledger.transfer.posted`, `.voided`, `.reversed` must be on `payment-sdk:0.3.4`+.

```java
// Before
Ufid transferId = event.transfer().id();
// After
Txid transferId = event.transfer().id();
```

Mechanical type changes; `Account.id` stays `Ufid`. See `summer-payment-sdk/references/versions/0.3.4.md`.

At the TigerBeetle CDC boundary, convert from raw u128 bytes or from a UUID-typed legacy field:

```java
Txid txid = Txid.from16Bytes(raw128);   // throws if upper 8 bytes != 0
Txid txid = Txid.fromUUID(ufid.toUUID());
```

## 3. Optional â€” adopt `Txid` for new transaction reference numbers

**Decide on the column type first.** New schemas should use `BIGINT NOT NULL PRIMARY KEY` (4Ă— smaller index, native int comparison). Legacy schemas standardized on `UUID` keep `UUID` and rely on `TxidUuidConverter`.

```sql
CREATE TABLE transactions (
    txid BIGINT NOT NULL PRIMARY KEY,
    ufid UUID   NOT NULL UNIQUE,         -- canonical internal key
    ...
);
```

Set the writer accordingly in `application.yml`:

```yaml
summer:
  r2dbc:
    txid-column-type: bigint   # default 'uuid' â€” UUID-typed legacy schemas keep the default
```

(Reads are unambiguous regardless of the setting; only the **writer** is exclusive. Mixing BIGINT and UUID `Txid` columns in the same app is unsupported on the write side â€” standardize on one. See `summer-data/references/versions/0.3.5.md`.)

**Wire `TxidGenerator`.** Pick the pattern that matches your fleet's identity model:

- **StatefulSet (zero config):** `MachineIdResolver.resolve()` reads `HOSTNAME=<svc>-<n>`.
- **Deployment, fixed identity:** inject `SUMMER_TXID_MACHINE_ID` per pod.
- **Deployment, autoscaled / Cloud Run:** `RedisMachineIdReservation` â€” self-healing slot lease.

See `summer-core/references/versions/0.3.5.md` for the full wiring snippets.

**Don't change `EPOCH_MS`** after first deploy (the custom epoch is a permanent decision). **Don't reuse retired pod ordinals** within 100 ms on scale-down (StatefulSets handle this naturally).

## 4. Optional â€” adopt `summer-file` for zip/xlsx exports

Most exporting services grew their own `ZipOutputStream` plumbing. Replace with one call:

```gradle
implementation 'io.f8a.summer:summer-file'
```

```java
ExportSpec<TxnRow> spec = ExportSpec.<TxnRow>builder()
    .baseName("transactions")
    .maxRowsPerFile(50_000)
    .maxZipSizeBytes(50L * 1024 * 1024)
    .chunkWriter(xlsxWriter)
    .build();

ZipExporter.writeZip(rows, out, spec);
// or
PipedInputStream in = ZipExporter.pipeZip(rowsFlux, spec,
    r -> Schedulers.boundedElastic().schedule(r));
storage.upload(bucket, key, in);
```

**Don't close the `OutputStream` from inside the `ChunkWriter`** â€” the exporter owns its lifecycle. Closing it early truncates the zip. See `summer-file/SKILL.md` and `summer-file/references/export-examples.md`.

## Verification

- A new transfer minted by `TxidGenerator.next()` round-trips through `LedgerTransferEvent` and a Kafka topic without losing precision.
- `Txid.from16Bytes(...)` survives the TigerBeetle CDC boundary.
- A `summer-file` export over an empty `Flux` still produces a zip with one entry (header row).
- `summer.r2dbc.txid-column-type=bigint` â€” startup log shows `TxidConverter.Writing` registered; `txid-column-type=uuid` â€” startup log shows `TxidUuidConverter.Writing`. Misconfigured values fail boot with a clear `BindException`.
