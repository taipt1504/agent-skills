# Migration: Summer 0.3.1 → 0.3.2

> **Multi-realm release.** Single-`keycloak.*` block + boolean `sync-role.enabled` are removed
> in this hop. Every issuer must be declared as an entry under `providers.<id>`. This is the
> **largest schema change** in the entire 0.x series for `summer-security`.

> Per-skill detail: [`summer-security/references/versions/0.3.2.md`](../../../summer-security/references/versions/0.3.2.md),
> [`summer-data/references/versions/0.3.2.md`](../../../summer-data/references/versions/0.3.2.md).

## Step 1 — security YAML reshape (BREAKING)

```yaml
# Before (0.3.0 → 0.3.1)
f8a.security.apisix.resource-server:
  keycloak:
    server-url: https://kc.example.com
    realm: master
    client-id: admin-cli
    client-secret: ${KC_SECRET}
  sync-role:
    enabled: true
  group-role-authorization:
    enabled: true
    claim-name: role_groups
    l1: { ttl: 60s }
    l2: { ttl: 5m, ... }

# After (0.3.2)
f8a.security.apisix.resource-server:
  providers:
    primary:                                       # provider id = scope key
      server-url: https://kc.example.com
      realm: master
      client-id: admin-cli
      client-secret: ${KC_SECRET}
      group-role-authorization: true               # per-provider opt-in
  sync-role: primary                               # provider-id pointer (not a boolean)
  group-role-authorization:                        # global cache config only
    claim-name: role_groups
    l1: { ttl: 60s }
    l2: { ttl: 5m, ... }
```

The map key (e.g. `primary`) is the **scope key**. Pick a stable id; downstream invalidation
calls use it as a string identifier.

## Step 2 — declare additional realms (if multi-tenant)

```yaml
providers:
  primary:
    server-url: https://kc.example.com
    realm: own
    client-id: ${KC_OWN_ID}
    client-secret: ${KC_OWN_SECRET}
    group-role-authorization: true
  partner:
    server-url: https://kc.partner.com
    realm: tenant1
    # No client-id/secret — partner only does inbound auth, no admin calls.
  vn-tenant:
    server-url: https://kc.example.com
    realm: tenant_vn
    client-id: ${KC_VN_ID}
    client-secret: ${KC_VN_SECRET}
    group-role-authorization: true
```

`MultiRealmAuthenticationConverter` routes inbound tokens by `iss` claim to the matching
provider. Tokens from issuers not in the map are rejected with 401.

## Step 3 — fail-fast guards (be aware)

Startup throws `IllegalStateException` when:

- `sync-role: <id>` references a provider id that isn't in the map.
- The targeted `sync-role` provider lacks `client-id` or `client-secret`.
- Any provider with `group-role-authorization: true` lacks admin credentials.

Fix the YAML; the error message names the offending provider id.

## Step 4 — direct callers of `GroupRoleResolver` / `GroupRoleInvalidator`

If you instantiate these classes in tests or custom wiring, signatures changed (BREAKING):

```java
// GroupRoleResolver — scopeKey is now first arg
new GroupRoleResolver("primary", Duration.ofSeconds(60), redis,
                      "auth-group-role:", Duration.ofMinutes(5), fetcher);

// GroupRoleInvalidator — invalidate now requires scopeKey
groupRoleInvalidator.invalidate("primary", "/admins").subscribe();
```

(0.3.3 adds a single-arg `invalidate(groupName)` broadcast overload — see the next migration.)

## Step 5 — external pub/sub publishers

Keycloak SPIs, ops scripts, and any other process publishing raw payloads to the Redis
invalidation channel must switch from bare `<groupName>` to `<scopeKey>:<groupName>`.

```diff
- PUBLISH group-role-changes "/admins"
+ PUBLISH group-role-changes "primary:/admins"
```

The 0.3.2 listener silently drops payloads without `:` separator (logged at WARN —
"Malformed invalidation payload").

## Step 6 — L2 Redis keys reshape (automatic)

Keys change from `<keyPrefix><groupName>` to `<keyPrefix><scopeKey>:<groupName>`. Existing
keys from 0.3.0/0.3.1 won't be hit by the new resolver — they TTL out (default 5m). No manual
deletion needed unless a long L2 TTL is configured (in which case `redis-cli --scan --pattern
'auth-group-role:*' | xargs redis-cli unlink` is the cleanest).

## Step 7 — outbox YAML cleanup (CDC users)

Two properties dropped silently (see `summer-data/references/versions/0.3.2.md`):

| Removed | Replacement |
|---|---|
| `f8a.outbox.publisher.cdc.bootstrap-servers` | `spring.kafka.bootstrap-servers` (always shared) |
| `f8a.outbox.publisher.cdc.additional-properties` | `spring.kafka.*` (single source of truth) |

Startup now fails with a clear message if `spring.kafka.bootstrap-servers` is missing under
CDC mode. Set it.

## Step 8 — verification

- [ ] App boots with `Created GroupRoleResolver scope='<id>' (iss=<issuer>)` log line for every
      opt-in provider.
- [ ] Token from a known issuer authenticates; token from an unknown `iss` returns 401 with
      body `Untrusted token issuer: <iss>`.
- [ ] L2 keys observed in Redis: `auth-group-role:<provider-id>:<group>`.
- [ ] (CDC) Debezium boots on a SASL-protected cluster without manual JAAS config in
      `f8a.outbox.publisher.cdc.*`.
- [ ] (Multi-tenant) cache invalidation in tenant A doesn't clear tenant B's L2 entries.

## Step 9 — plan 0.3.3 (optional)

0.3.3 (Unreleased) adds two **purely additive** features that simplify the multi-realm story
further:

- Optional `provider.issuer-uri` — split internal Admin URL from public token issuer.
- `GroupRoleInvalidator.invalidate(groupName)` broadcast overload — caller doesn't need to
  know the provider topology.

See [`0.3.2-to-0.3.3.md`](0.3.2-to-0.3.3.md) for the (small) checklist.
