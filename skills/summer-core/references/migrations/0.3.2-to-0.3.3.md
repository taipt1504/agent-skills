# Migration: Summer 0.3.2 → 0.3.3

> 0.3.3 released 2026-05-07. Both features are **purely additive**. Existing 0.3.2 configs and
> code continue to behave identically. No required steps — adopt only when needed.

> Per-skill detail: [`summer-security/references/versions/0.3.3.md`](../../../summer-security/references/versions/0.3.3.md).

## Step 1 (optional) — set `provider.issuer-uri` when you need a split URL

Use when the public token issuer differs from the URL the application uses to call the Admin
API. Common in production with reverse-proxy or internal/external hostname split.

```yaml
f8a.security.apisix.resource-server:
  providers:
    backoffice:
      server-url: http://keycloak.internal:8080                  # Admin API (internal)
      realm: ewallet_backoffice
      issuer-uri: https://auth.example.com/realms/ewallet_backoffice  # Token iss (public)
      client-id: ${KC_CLIENT_ID}
      client-secret: ${KC_CLIENT_SECRET}
      group-role-authorization: true
```

Leave `issuer-uri` blank/null to keep the computed fallback `serverUrl + "/realms/" + realm`.
Trailing slashes on either source are normalised away.

## Step 2 (optional) — adopt broadcast invalidation

When the publisher can't (or shouldn't) map realm → provider id, use the single-arg overload:

```java
// 0.3.2 — must know scope
groupRoleInvalidator.invalidate("backoffice", "/admins").subscribe();

// 0.3.3 — broadcast across all providers
groupRoleInvalidator.invalidate("/admins").subscribe();
```

The broadcast publishes payload `*:/admins`; the listener iterates every registered resolver.
Useful for:

- Generic CLI / cron tools clearing a known group.
- Keycloak SPIs that can't resolve a stable provider-id mapping.
- Cross-tenant rebuilds (rare).

Prefer the targeted form when the caller knows the provider — broadcast may evict unrelated
cache entries that happen to share the same group name across providers.

External raw `PUBLISH` form:

```
PUBLISH group-role-changes "*:/admins"            # broadcast
PUBLISH group-role-changes "backoffice:/admins"   # targeted (still works)
```

## Breaking changes

None. 0.3.3 is fully backward compatible with 0.3.2.

## Verification

- (If `issuer-uri` set) startup log shows `Created GroupRoleResolver scope='<id>' (iss=<issuer-uri>)`
  with the override value (not the computed value).
- (If broadcast adopted) `redis-cli MONITOR` shows `PUBLISH group-role-changes *:<group>`.
- Listener log `Broadcasting invalidation for group '<group>'` appears when wildcard scope is
  received.
