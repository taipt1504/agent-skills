# Migration: Summer 0.3.3 → 0.3.4

Focused security release. Two concrete actions for service maintainers; everything else is internal cleanup.

## 1. Bump the BOM

```gradle
implementation platform('io.f8a.summer:summer-platform:0.3.4')
```

`paymentSdkVersion` stays at `0.3.3` (the BOM pins this; you don't override unless you have a reason). No DB schema changes.

## 2. Delete service-local `SseQueryParamTokenFilter` copies

Browsers' `EventSource` API can't send custom headers, so SSE endpoints authenticate via `?token=<jwt>`. Summer now ships the filter — services that had drifted copies (e.g. `wallet-ms`, `core-ledger-ms` under `config/sse/`) must delete the local copy and register a `SseAuthCustomizer` bean instead.

```java
// BEFORE — service-local class. Delete it.
@Component
public class SseQueryParamTokenFilter implements WebFilter { ... }

// AFTER — Summer's filter is auto-configured. Just declare the binding.
@Bean
SseAuthCustomizer mySseAuth() {
  return reg -> reg
      .pathStartsWith("/api/v1/public/sse/admin/", "bo")        // 'bo' tokens only
      .pathStartsWith("/api/v1/public/sse/",       "end-user")  // 'end-user' tokens only
      .acceptEventStream("end-user");                            // catch-all SSE → end-user
}
```

**Provider ids must exist** in `f8a.security.apisix.resource-server.providers.<id>` — auto-config throws `IllegalStateException` at startup if a customizer references an unknown id, listing the known ones. Catches typos before they 401 every SSE request silently.

**Keep your `permitAll()` rules** for SSE paths:

```java
.pathMatchers(HttpMethod.GET, "/api/v1/public/sse/**").permitAll()
```

The SSE filter populates the security context before the APISIX filter ever sees the request, but `permitAll()` is what lets the request through `authorizeExchange()` for token-based auth via the SSE path.

The pre-Summer filters typically hardcoded a provider id (`provider_a`) or called `apisixProperties.getKeycloak()` against the long-removed 0.3.2 single-provider shape. Both are cleaned up by deleting the file.

## 3. Verify `KeycloakRoleSynchronizer` injection sites

The bean is now conditionally created via `@ConditionalOnExpression("!'${f8a.security.apisix.resource-server.sync-role:}'.trim().isEmpty()")` — when `sync-role` is unset, the bean is **not registered at all** (was: registered as a null bean).

- `@Autowired(required = false) KeycloakRoleSynchronizer` — same observable behaviour (returns null when unsatisfied). No change.
- `ObjectProvider<KeycloakRoleSynchronizer>` — same observable behaviour. No change.
- `@Autowired KeycloakRoleSynchronizer` (required, sync disabled) — was: `NullBean` injected → NPE later. Now: clean `NoSuchBeanDefinitionException` at startup. Either drop the dependency or set `sync-role: <providerId>` in config.

**Operator note:** the `"sync-role not configured — role synchronization disabled"` info log is gone. Absence of the `"Role synchronization will run against provider..."` line at startup is now the signal that sync is off.

## 4. (Optional) Use `ProviderJwtDecoderResolver` for custom auth filters

If you have a service-local filter that maps `provider key → ReactiveJwtDecoder`, replace it with the bean Summer now auto-wires:

```java
@RequiredArgsConstructor
@Component
public class CustomAuthFilter implements WebFilter {
  private final ProviderJwtDecoderResolver decoderResolver;

  @Override
  public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
    String providerId = pickProvider(exchange);
    ReactiveJwtDecoder decoder = decoderResolver.apply(providerId);
    // decoder is null if providerId is unknown — treat as misconfiguration
  }
}
```

Override via your own `ProviderJwtDecoderResolver` bean only for tests where you need a static decoder map.

## Verification

- Boot-time log shows `Created GroupRoleResolver scope='<id>' (iss=...)` per provider.
- A service that had a local SSE filter no longer ships one; an `SseAuthCustomizer` bean exists; SSE handshake against `?token=<jwt>` still succeeds.
- Setting an SSE rule to reference a typo'd provider id fails the boot with a clear `IllegalStateException` naming both the typo and the valid ids.
- Disabling `sync-role` (omitting the property) and `@Autowired KeycloakRoleSynchronizer` (required) fails fast with `NoSuchBeanDefinitionException`.
