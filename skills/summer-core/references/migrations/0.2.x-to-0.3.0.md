# Migration: Summer 0.2.x → 0.3.0

> Cross-cutting migration guide. Per-skill detail lives in each skill's `versions/` overlays.
> Use this guide when bumping from any 0.2.x line to 0.3.0.

## What's affecting your service

If you're on **0.2.4 → 0.2.9**, the upgrade to 0.3.0 is **small** for everything except direct
`ApisixAuthenticationManager` instantiation:

- New: JWT blacklist via Redis (opt-in via `blacklist-prefix-key`).
- New: `JwtBlacklistChecker` strategy bean.
- Breaking: `ApisixAuthenticationManager` constructor now 3-arg.

If you're skipping over `0.2.4 → 0.2.9`, also read:
- [`summer-security/references/versions/0.2.4.md`](../../../summer-security/references/versions/0.2.4.md) — Keycloak shared block consolidation, group-role authorization, `Mono`-returning converter (BREAKING).
- [`summer-data/references/versions/0.2.8.md`](../../../summer-data/references/versions/0.2.8.md) — Outbox CDC mode, `OutboxProperties` redesign, Ufid-typed event id.
- [`summer-core/references/versions/0.2.5.md`](../../../summer-core/references/versions/0.2.5.md) and `0.2.6.md`, `0.2.8.md` — UFID introduction → Crockford Base32 → encoding cleanup.

## Step 1 — bump platform BOM

```diff
- implementation platform('io.f8a.summer:summer-platform:0.2.9')
+ implementation platform('io.f8a.summer:summer-platform:0.3.0')
```

Module dependencies stay at the same coordinates — no renames in this transition.

## Step 2 — fix direct `ApisixAuthenticationManager` instantiation

The 1-arg and 2-arg constructors are removed. If you instantiate the manager directly in tests
or custom security wiring, pass three args:

```java
// Before (0.2.x)
new ApisixAuthenticationManager(mapper);
new ApisixAuthenticationManager(mapper, converter);

// After (0.3.0)
new ApisixAuthenticationManager(mapper, converter, blacklistChecker);
```

`blacklistChecker` may be `null` to disable the check. Spring auto-config handles this
transparently — production code wiring through auto-configuration is unchanged.

## Step 3 (optional) — enable JWT blacklist

```yaml
f8a.security.apisix.resource-server:
  blacklist-prefix-key: "auth-blacklist"
```

Requires Redis on the classpath. Publish a revocation:

```
SET auth-blacklist:<jti> 1 EX <remaining-token-seconds>
```

The next request with that `jti` returns 401. Behaviour fails open if Redis is unavailable.

To use a non-Redis backend, provide a `JwtBlacklistChecker` bean
(`@ConditionalOnMissingBean` skips the default).

## Step 4 — verification checklist

- [ ] App boots without `BeanInstantiationException` on `ApisixAuthenticationManager`.
- [ ] Existing `@PreAuthorize` paths still authenticate.
- [ ] If blacklist enabled: revoking a token in Redis blocks the next request with that `jti`.
- [ ] If Redis is briefly unavailable, requests still succeed (fail-open).

## Step 5 — when ready, plan the 0.3.2 jump

0.3.0 still uses the **single-`keycloak.*` block** schema. 0.3.2 replaces it with
`providers.<id>:` (multi-realm). When you're ready, read
[`0.3.0-to-0.3.2.md`](0.3.0-to-0.3.2.md) — that hop is much larger.
