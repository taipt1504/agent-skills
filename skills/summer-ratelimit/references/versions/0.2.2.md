# Summer 0.2.2 — summer-ratelimit (module added)

Released: 2026-03-04

The `summer-ratelimit-core` and `summer-ratelimit-autoconfigure` modules are introduced in this
version. Earlier Summer versions had no rate-limiting module at all.

## Features

### Three rate-limiting strategies

- **Fixed Window** — bucket per discrete window, simplest and cheapest.
- **Sliding Window** — log-based, smooths burst boundaries.
- **Token Bucket** — refill rate + capacity, allows controlled bursts.

### Per-scope policies

Different business domains can use different strategies and limits via
`f8a.rate-limiter.policies.<scope>.*`. A single global `key-prefix` is shared across all
policies.

```yaml
f8a:
  rate-limiter:
    enabled: true
    key-prefix: "rl:"
    policies:
      api-default:
        strategy: fixed-window
        limit: 100
        window: 1m
      checkout:
        strategy: token-bucket
        capacity: 50
        refill-rate: 10
        refill-period: 1s
```

### Storage backends

- **Redis** (default) — uses atomic Lua scripts. Auto-detected when
  `spring-boot-starter-data-redis-reactive` is on the classpath.
- **In-memory** — fallback for single-instance / tests.

### `RateLimiterService` API

```java
// Manual handling — caller decides what to do with denial.
rateLimiterService.tryAcquire(key, limit, window)
    .flatMap(result -> result.allowed()
        ? proceed()
        : Mono.error(new TooManyException(result.resetAt())));

// Auto-throw — denial maps to 429 with limit + resetAt in error details.
rateLimiterService.acquire(key)
    .then(proceed());
```

## Breaking changes

None — this is an additive new-module release.

## Documentation

New `docs/HOW_WE_RATELIMIT.md` in Summer covering setup, configuration, and usage patterns.

## Migration from 0.2.1

Add the dependency:

```gradle
implementation 'io.f8a.summer:summer-ratelimit-autoconfigure'
```

(Pick this up via the BOM — no version pin needed.) Configure at least one policy under
`f8a.rate-limiter.policies.*`. Inject `RateLimiterService` and call `acquire()` /
`tryAcquire()` on rate-limited paths.

## Verification

- `RateLimiterService` bean present in `ApplicationContext`.
- Redis Lua scripts execute (visible in `redis-cli MONITOR`).
- 429 response carries `limit` and `resetAt` in `details`.
