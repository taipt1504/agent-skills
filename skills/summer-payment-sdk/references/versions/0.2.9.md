# Summer 0.2.9 — summer-payment-sdk

Released: 2026-04-14

Event classes split into domain subpackages — BREAKING import path change. `CustomerInfoEvent` expanded with a `CustomerInfoEventType` enum.

## Breaking changes

### Event subpackages

`io.f8a.summer.payment.event.*` is now split:

| Sub-package | Events |
|---|---|
| `event.ledger` | `LedgerTransferEvent`, `EntriesPostedEvent`, `EntriesRejectedEvent`, `PostEntriesCommand`, `LedgerEventType`, `SagaRejectionCode` |
| `event.bank` | `BankNotifyTransaction*` and related |
| `event.party` | `CustomerInfoEvent`, `CustomerInfoEventType`, party lifecycle |
| `event.va` | Virtual-account events |
| `event.wallet` | Wallet lifecycle / status |

Imports must be updated:

```java
// Before
import io.f8a.summer.payment.event.LedgerTransferEvent;
// After
import io.f8a.summer.payment.event.ledger.LedgerTransferEvent;
```

The wire JSON shape is unchanged — only Java import paths change. Consumers reading the same topic with the old import will fail to compile; the wire bytes are still compatible across the rename.

## Features

### `CustomerInfoEventType` enum

Discriminator for `CustomerInfoEvent` payloads, used by the party subpackage. Lets a single Kafka topic carry multiple sub-event shapes with one consumer.

## Migration from 0.2.8

1. Run a search-replace across imports:
   - `io.f8a.summer.payment.event.LedgerTransferEvent` → `io.f8a.summer.payment.event.ledger.LedgerTransferEvent`
   - `io.f8a.summer.payment.event.EntriesPostedEvent` → `io.f8a.summer.payment.event.ledger.EntriesPostedEvent`
   - …and so on for each event class.
2. Confirm compilation; the wire format is unchanged, so deployed consumers don't need a rolling restart.

## Verification

- Source files compile against the new package layout.
- Wire payloads round-trip identically — a JSON string serialized by 0.2.8 deserializes against 0.2.9 with no field changes.
