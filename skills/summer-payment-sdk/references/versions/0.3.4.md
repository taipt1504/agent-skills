# Summer payment-sdk 0.3.4 — Transfer ids become `Txid`

Released: 2026-05-10 (paired with `summer-platform:0.3.5`)

Transfer ids on `LedgerTransferEvent.Transfer` are now `Txid` (was `Ufid`). BREAKING wire-format change for every consumer of `ledger.transfer.posted` and siblings.

## Breaking changes

### `LedgerTransferEvent.Transfer.id` / `pendingId` are now `Txid`

The wire JSON for those fields changes from a UUID string to the 18-digit zero-padded decimal produced by `Txid.toJsonValue()`. Older consumers reading the field as `Ufid` / `UUID` will fail deserialization.

`Account.id` stays `Ufid` — TigerBeetle accounts can predate the Txid cutoff and are not all generated by `TxidGenerator`.

**Conversion at the TigerBeetle CDC boundary:**

```java
// From the Ufid the CDC stream carries on accounts that predate Txid:
Txid txid = Txid.fromUUID(ufid.toUUID());   // throws if upper 8 bytes != 0

// Or from raw u128 bytes:
Txid txid = Txid.from16Bytes(rawBytes);
```

Both `from*` constructors throw if the upper half of the 128-bit value is non-zero. A `Txid` produced by `TxidGenerator` always has the upper 64 bits zero — a non-zero half means the id came from elsewhere (legacy, manual insert, foreign service) and is a bug worth surfacing immediately.

## Features

No new annotations or DTOs in this release — it is a single, focused type bump on the existing `Transfer` record.

## Migration from 0.3.3

1. Bump `summer-payment-sdk` to `0.3.4` (the `summer-platform:0.3.5` BOM brings it transitively).
2. Walk type errors — every call site that took `Ufid` from a `Transfer` field now takes `Txid`. Most diffs are mechanical:
   ```java
   // Before
   Ufid transferId = event.transfer().id();
   // After
   Txid transferId = event.transfer().id();
   ```
3. **Persistence:** if you store the transfer id in a `UUID` column, R2DBC's `TxidUuidConverter` (auto-registered, default `summer.r2dbc.txid-column-type=uuid`) packs the `Txid` into a UUID with `MSB=0`. Greenfield schemas can flip to `BIGINT` via `summer.r2dbc.txid-column-type=bigint` and the new column type — see `summer-data/references/versions/0.3.5.md`.
4. Producers that **mint** ids: switch from `Ufid.generate()` to `TxidGenerator.next()`. See `summer-core/references/versions/0.3.5.md` for the generator wiring (StatefulSet vs Redis lease).
5. Consumers reading `event.transfer().id()` for display: `Txid.toString()` is the 18-digit decimal — pass it through unchanged. Don't reformat it as a UUID.

## Verification

- A test event with `{"id": "000123456789012345"}` deserializes; the same with `{"id": "<uuid>"}` fails fast.
- `Txid.from16Bytes(...)` round-trips through TigerBeetle CDC without losing precision.
- All non-Transfer ids (e.g. `Account.id`) still serialize as UUID strings — confirm Jackson didn't accidentally swap a field type.
