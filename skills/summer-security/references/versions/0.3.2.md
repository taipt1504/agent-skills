# Summer 0.3.2 — summer-security

Released: 2026-04-20

This is the **multi-realm** release. Single-tenant `keycloak.*` config and boolean
`sync-role.enabled` are gone — every issuer must be declared as an entry under `providers.<id>`.
Group-role caches are now scoped per provider.

## Breaking changes

### Schema: single-keycloak → `providers.<id>` map

```yaml
# Before (0.2.4 → 0.3.0)
f8a.security.apisix.resource-server:
  keycloak:
    server-url: https://kc.example.com
    realm: master
    client-id: admin-cli
    client-secret: secret
  sync-role:
    enabled: true
  group-role-authorization:
    enabled: true

# After (0.3.2)
f8a.security.apisix.resource-server:
  providers:
    primary:
      server-url: https://kc.example.com
      realm: master
      client-id: admin-cli
      client-secret: secret
      group-role-authorization: true       # opt-in per provider
  sync-role: primary                       # provider id pointer (not a boolean)
  group-role-authorization:                # global cache config only
    claim-name: role_groups
    l1: { ttl: 60s }
    l2:
      ttl: 5m
      invalidation-channel: "group-role-changes"
      key-prefix: "auth-group-role:"
```

The map key (e.g. `primary`) is the **scope key** used downstream by the resolver and pub/sub
payload (`<scope>:<group>`).

### `MultiRealmAuthenticationConverter` replaces `TrustedIssuerConverter`

Inbound tokens are routed to a per-issuer converter by `MultiRealmAuthenticationConverter` based
on the `iss` claim. Tokens from issuers not in `providers.*` are rejected with 401.

```java
// Inbound flow
MultiRealmAuthenticationConverter
  └─ token.iss → byIssuer.get(iss) → per-provider converter
       └─ provider has group-role-authorization: true ? GroupRoleAuthenticationConverter
                                                       : UserInfoAuthenticationConverter
```

### `sync-role` is now a top-level provider-id pointer

Previous boolean opt-in is removed. `sync-role: <provider-id>` selects exactly one provider that
receives `@AuthRoles` → Keycloak sync at startup. Leave blank/null to disable.

```yaml
sync-role: backoffice                    # 0.3.2+: provider id pointer
# vs.
sync-role:                                # 0.2.4 → 0.3.0
  enabled: true
```

Startup fails fast when:
- `sync-role` references a provider id that isn't in the map.
- The targeted provider lacks `client-id` or `client-secret`.

### `group-role-authorization` is now per-provider opt-in

The top-level `enabled: true` flag is replaced by a per-provider `group-role-authorization: true`
flag. The global block keeps cache-tuning settings only (`claim-name`, `l1`, `l2`).

A provider with `group-role-authorization: true` must have admin credentials (else startup
fails).

### Group-role cache scoped by provider id

Each `GroupRoleResolver` now carries a `scopeKey` (= provider id). L2 Redis keys use the form
`<keyPrefix><scopeKey>:<groupName>`, so multiple providers can share one Redis instance even
when group names collide. Pub/sub payload format becomes `<scopeKey>:<groupName>` — the listener
matches the `scopeKey` against the resolver and clears only the matching realm's cache.

`GroupRoleResolver` constructor signature changed (BREAKING for direct instantiation in
tests/custom wiring):

```java
// Before
new GroupRoleResolver(Duration.ofSeconds(60), redis, "auth-group-role:", Duration.ofMinutes(5), fetcher);

// After
new GroupRoleResolver("backoffice", Duration.ofSeconds(60), redis, "auth-group-role:", Duration.ofMinutes(5), fetcher);
```

`GroupRoleInvalidator.invalidate(...)` becomes 2-arg `invalidate(scopeKey, groupName)`.
External publishers that pushed bare `groupName` payloads must prefix with a scope.

### L2 Redis key shape changed

From `<keyPrefix><groupName>` to `<keyPrefix><scopeKey>:<groupName>`. Old keys won't be hit by
the new resolver — they TTL out (default 5m). No manual migration is required unless a long L2
TTL is configured.

## Features

- New `Provider` properties: `serverUrl`, `realm`, `client-id`, `client-secret`,
  `group-role-authorization` flag, fail-fast guard `hasAdminCredentials()`.
- `ReactiveApisixKeycloakAdminAutoConfiguration` builds one resolver per provider that opts in,
  plus a single pub/sub listener that routes by `scopeKey`.

## Migration from 0.3.0

1. **Restructure YAML**: lift the `keycloak.*` block under `providers.<id>:` (chosen id becomes
   the scope key), set the per-provider `group-role-authorization: true` flag, and convert
   `sync-role.enabled: true` → `sync-role: <provider-id>`.
2. **Update direct `GroupRoleResolver` / `GroupRoleInvalidator` callers** to pass `scopeKey`.
3. **External pub/sub publishers** (Keycloak SPI, ops scripts): switch payload from
   `<groupName>` to `<scopeKey>:<groupName>`.
4. **Tests**: any mock for `MultiRealmAuthenticationConverter` should construct
   `Map<String, Converter<...>>` keyed by issuer URI.

See [`summer-core/references/migrations/0.3.0-to-0.3.2.md`](../../../summer-core/references/migrations/0.3.0-to-0.3.2.md)
for the step-by-step guide.

## Verification

- Log line `Created GroupRoleResolver scope='<id>' (iss=<issuer>)` for every opt-in provider.
- Log line `Starting group-role invalidation listener on channel '<channel>'` when L2 Redis is
  configured.
- Token from an unknown `iss` returns 401 with body `Untrusted token issuer: <iss>`.
- L2 keys observed in Redis: `auth-group-role:<provider-id>:<group>`.
