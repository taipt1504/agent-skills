# Summer 0.3.3 — summer-security

Released: 2026-05-07

## Features

### Optional `provider.issuer-uri`

Decouples the JWT issuer (matched against the inbound token's `iss` claim) from `server-url` (the
Admin API base URL). Useful when tokens are issued at a public hostname while the Admin API is
reachable only via an internal hostname — common in production with a reverse proxy or
internal/external split.

```yaml
f8a.security.apisix.resource-server:
  providers:
    backoffice:
      server-url: http://keycloak.internal:8080            # Admin API (internal)
      realm: ewallet_backoffice
      issuer-uri: https://auth.example.com/realms/ewallet_backoffice  # Token iss (public)
      client-id: ${KC_CLIENT_ID}
      client-secret: ${KC_CLIENT_SECRET}
      group-role-authorization: true
```

When `issuer-uri` is `null` or blank, the resolver falls back to the existing computed value
`serverUrl + "/realms/" + realm`. Trailing slashes on either source are normalised away.
**Purely additive** — no migration needed for existing configs.

### Broadcast invalidation in `GroupRoleInvalidator`

New single-arg overload `invalidate(String groupName)` publishes a wildcard `*:<group>` payload
that clears the matching group entry across **every** registered resolver. Useful for callers
that don't (or shouldn't) know the realm topology — admin tools, generic CLIs, Keycloak SPIs
that can't map realm → provider id.

```java
// Targeted (preferred when caller knows the provider).
groupRoleInvalidator.invalidate("backoffice", "/admins").subscribe();

// Broadcast — clears "/admins" across all providers' caches. (NEW in 0.3.3)
groupRoleInvalidator.invalidate("/admins").subscribe();
```

Constant `GroupRoleInvalidator.BROADCAST_SCOPE = "*"` is exposed for tests / direct callers.
The listener detects the wildcard scope and iterates every resolver instead of searching for a
single match.

When raw-`PUBLISH` from outside Java (Keycloak SPI, cron tools), use the same payload format:

```
PUBLISH group-role-changes "backoffice:/admins"   # targeted
PUBLISH group-role-changes "*:/admins"            # broadcast
```

### When to use which form

| Scenario | Form | Reason |
|---|---|---|
| Admin service mutating role groups in **its own** realm only | targeted | Avoids thrashing other providers' caches when group names collide |
| Generic CLI / cron clearing a known group | broadcast | Caller doesn't have a provider-id mapping; one publish reaches all matching scopes |
| Keycloak SPI listening to admin events | targeted (preferred) — broadcast acceptable | SPI knows realm → map realm to provider id |
| Cross-tenant rebuild (unusual) | broadcast | Force a uniform refresh across every scope |

## Breaking changes

None. Both additions are backward compatible.

## Migration from 0.3.2

Optional. To adopt:

1. **Custom issuer URI** — set `provider.issuer-uri` only when you need the split. Existing
   configs without the field continue to behave identically.
2. **Broadcast invalidator** — replace `groupRoleInvalidator.invalidate(scope, group)` with
   `groupRoleInvalidator.invalidate(group)` only when the caller cannot reliably identify the
   scope. Prefer targeted otherwise to avoid unnecessary cache thrash.

## Verification

- Setting `issuer-uri` on a provider and inspecting startup log:
  `Created GroupRoleResolver scope='<id>' (iss=<issuer-uri>)` — the logged issuer matches the
  override, not the computed value.
- Calling `groupRoleInvalidator.invalidate(group)` produces a Redis publication with payload
  starting with `*:` (visible via `redis-cli MONITOR`).
- Listener log `Broadcasting invalidation for group '<group>'` when wildcard scope is received.
