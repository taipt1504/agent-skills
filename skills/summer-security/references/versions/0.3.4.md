# Summer 0.3.4 — summer-security

Released: 2026-05-08

A focused security release. **SSE auth lands in Summer** as `SseQueryParamTokenFilter` — promoted from copies that were drifting in `wallet-ms` and `core-ledger-ms`. Authentication is driven by an `SseAuthCustomizer` that binds request paths to a **provider key** (not the JWT's `iss` claim), and the per-provider decoder is now its own reusable bean (`ProviderJwtDecoderResolver`). The role-sync bean is also tidied: it's no longer registered as a null bean when disabled.

## Features

### `SseQueryParamTokenFilter` in `summer-apisix-resource-server`

Browsers' `EventSource` API can't send custom headers, so SSE endpoints authenticate via `?token=<jwt>`. Summer now ships the filter; remove copies from your service.

**Package:** `io.f8a.summer.security.apisix.server.resource.web.server.filter.SseQueryParamTokenFilter`

**Wiring:** auto-configured as a `WebFilter` bean by `ReactiveApisixResourceServerAutoConfiguration` at order `-100` (runs before the APISIX `AuthenticationWebFilter`). With **no** `SseAuthCustomizer` bean, the filter has zero rules and is a no-op — services opt in by registering a customizer.

**Routing model:** rules carry a **provider key** (one of `f8a.security.apisix.resource-server.providers.<id>`), not an issuer URL. The auto-config builds the `ReactiveJwtDecoder` against that provider's `issuerUri()`; the decoder enforces `iss` natively, so a token issued by a different registered provider can never authenticate on a path bound to another provider.

```java
@Bean
SseAuthCustomizer sseRoutes() {
  return reg -> reg
      .pathStartsWith("/api/v1/public/sse/admin/", "bo")        // only 'bo' tokens
      .pathStartsWith("/api/v1/public/sse/",       "end-user")  // only 'end-user' tokens
      .acceptEventStream("end-user");                            // any other SSE → end-user
}
```

Rules tested in registration order; first match wins. A rule that matches applies the filter; if no rule matches, the request bypasses it (downstream auth still runs).

**Public API:**

| Type | Package |
|---|---|
| `SseQueryParamTokenFilter` | `io.f8a.summer.security.apisix.server.resource.web.server.filter` |
| `SseQueryParamTokenFilter.MatchRule` | nested record `(Predicate<ServerWebExchange> matcher, String providerId)` — both required |
| `SseAuthCustomizer` | `io.f8a.summer.autoconfigure.security.apisix.resource.reactive` (functional, single `customize`) |
| `SseAuthCustomizer.Registration` | inner class with `matcher`, `pathStartsWith`, `pathContains`, `acceptEventStream` builders |

**Fail-fast at startup:**

- Customizer rule references a provider id not in `providers.<id>` → `IllegalStateException` listing the known provider ids. Catches a typo before it 401s every SSE request silently.
- Constructor of `MatchRule` rejects `null`/blank `providerId` and `null` `matcher` (defensive — populated via the builder methods, but a custom customizer that constructs `MatchRule` directly gets an early failure).

**Per-request flow:**

```
SSE request → first matching rule → providerId
            → decoderResolver.apply(providerId)
            → decoder.decode(token)            // verifies signature + iss
            → MultiRealmAuthenticationConverter.convert(oidcIdToken)
            → ReactiveSecurityContextHolder.withAuthentication(auth)
            → continue chain
```

The filter reuses the existing `MultiRealmAuthenticationConverter` (same one the `ApisixAuthenticationManager` uses), so the post-decode role/userinfo mapping is identical to non-SSE routes.

### `ProviderJwtDecoderResolver` — reusable provider-keyed decoder lookup

Extracted from `ReactiveApisixResourceServerAutoConfiguration`'s SSE bean into its own type so anything that needs `provider key → ReactiveJwtDecoder` can inject it.

**Package:** `io.f8a.summer.security.apisix.server.resource.authentication.ProviderJwtDecoderResolver`

```java
public class ProviderJwtDecoderResolver implements Function<String, ReactiveJwtDecoder> {
  private final ApisixResourceServerProperties properties;
  private final Map<String, ReactiveJwtDecoder> decodersMap = new ConcurrentHashMap<>();

  @Override
  public ReactiveJwtDecoder apply(String providerId) {
    Provider provider = properties.getProviders().get(providerId);
    if (provider == null) {
      return null;
    }
    return decodersMap.computeIfAbsent(
        providerId, id -> ReactiveJwtDecoders.fromIssuerLocation(provider.issuerUri()));
  }
}
```

**Behaviour:**

- Decoders are built **lazily** on first lookup via `ReactiveJwtDecoders.fromIssuerLocation(provider.issuerUri())` (synchronous HTTP fetch of `.well-known/openid-configuration`). Lazy-init means an unreachable OIDC server doesn't block app startup.
- Cached in a `ConcurrentHashMap` keyed by provider id.
- Returns `null` when the provider id is not in the map. Callers should treat null as a misconfiguration — auto-config validates rule provider ids at startup, so production hits are unexpected.

**Auto-configured bean:**

```java
@Bean
@ConditionalOnMissingBean
public ProviderJwtDecoderResolver providerJwtDecoderResolver(
    ApisixResourceServerProperties properties) {
  return new ProviderJwtDecoderResolver(properties);
}
```

Override by declaring your own `ProviderJwtDecoderResolver` bean (e.g. for tests where you need a static decoder map).

### `KeycloakRoleSynchronizer` — no more null `@Bean`

Previously the bean method returned `null` when `sync-role` wasn't configured, registering a `NullBean` placeholder in the application context. Functional but surprising — any `@Autowired KeycloakRoleSynchronizer` blew up with NPE-flavoured errors.

**Now:** the bean is conditionally created via `@ConditionalOnExpression("!'${f8a.security.apisix.resource-server.sync-role:}'.trim().isEmpty()")`, so when `sync-role` is unset the bean is **not registered at all**. Plus `@ConditionalOnMissingBean` so consumers can override.

```java
@Bean
@ConditionalOnMissingBean
@ConditionalOnExpression("!'${f8a.security.apisix.resource-server.sync-role:}'.trim().isEmpty()")
public KeycloakRoleSynchronizer keycloakRoleSynchronizer(...) {
  Provider syncTarget = findSyncTarget(props);   // throws on bad config; never null
  ...
}
```

`findSyncTarget` was simplified — the null-return for blank `sync-role` is gone (the condition handles it). It now only throws `IllegalStateException` on misconfig (unknown provider id, or provider missing admin credentials).

**Operator note:** the `"sync-role not configured — role synchronization disabled"` info log is gone. Absence of the `"Role synchronization will run against provider..."` line at startup is now the signal that sync is off.

## Breaking changes

### SSE filter copies in services should be removed

If your service has a `SseQueryParamTokenFilter` under `config/sse/` (e.g. `wallet-ms` and `core-ledger-ms` had drifted copies), delete it and add an `SseAuthCustomizer` bean instead. The Summer-provided filter is auto-wired, so a duplicate `@Component` class on the classpath either collides or silently shadows the Summer one.

```java
// Before — service-local filter
@Component
public class SseQueryParamTokenFilter implements WebFilter { ... }

// After — Summer's filter is auto-configured; just declare the binding
@Bean
public SseAuthCustomizer walletSseAuth() {
  return reg -> reg.pathStartsWith("/api/v1/public/sse/", "end-user");
}
```

The pre-Summer filters typically hardcoded a provider id (`provider_a`) or worse, called `apisixProperties.getKeycloak()` against the long-removed 0.3.2 single-provider shape. Both are cleaned up by deleting the file.

### `KeycloakRoleSynchronizer` no longer registered as a null bean

If anything used `ObjectProvider<KeycloakRoleSynchronizer>` and relied on `getIfAvailable()` returning `null` when sync was disabled, the behaviour hasn't changed (`getIfAvailable()` still returns `null`). But code that `@Autowired KeycloakRoleSynchronizer` directly with `required = false` and checked for null will now see a true missing bean — `@Autowired(required = false)` returns `null` for unsatisfied dependency, which is the same observable behaviour, so no migration needed.

If you `@Autowired KeycloakRoleSynchronizer` (required=true) and sync was disabled, you previously got a `NullBean` injected and an NPE later. Now you get a clean `NoSuchBeanDefinitionException` at startup — fail earlier, fail clearer.

## Detection signals (for choosing this version)

You're on (or migrating to) 0.3.4 if you see any of:

- `io.f8a.summer.security.apisix.server.resource.web.server.filter.SseQueryParamTokenFilter` imported from Summer (not from a service-local package)
- `io.f8a.summer.autoconfigure.security.apisix.resource.reactive.SseAuthCustomizer` bean declarations
- `io.f8a.summer.security.apisix.server.resource.authentication.ProviderJwtDecoderResolver` imported or bean
- `@ConditionalOnExpression` with `f8a.security.apisix.resource-server.sync-role` on `keycloakRoleSynchronizer` (instead of returning `null` from the method body)
- `local.<service>.yml` or `application.yml` using `providers.<id>.*` with the **0.3.3 shape** unchanged (this release didn't reshape config — only added new beans)

## Migration from 0.3.3

1. Bump Summer version: `io.f8a.summer:summer-platform:0.3.4`
2. If your service has its own `SseQueryParamTokenFilter` under `config/sse/` (or similar): **delete it**, then add an `SseAuthCustomizer` bean that binds your SSE paths to the right provider id from `providers.<id>`:
   ```java
   @Bean
   SseAuthCustomizer mySseAuth() {
     return reg -> reg.pathStartsWith("/api/v1/public/sse/", "end-user");
   }
   ```
3. Verify the provider id you pass to the customizer exists in `f8a.security.apisix.resource-server.providers.<id>` — auto-config throws `IllegalStateException` at startup if it doesn't, listing the known ids.
4. If you have `securityFilterChain` rules that `permitAll()` your SSE paths (so the APISIX filter doesn't reject them as missing `X-Userinfo`), keep them — the SSE filter populates the security context before the APISIX filter ever sees the request, but the `permitAll()` is what lets the request through `authorizeExchange()` for token-based auth via the SSE path. Standard pattern:
   ```java
   .pathMatchers(HttpMethod.GET, "/api/v1/public/sse/**").permitAll()
   ```
5. Nothing to do for `KeycloakRoleSynchronizer` unless your service was doing `@Autowired KeycloakRoleSynchronizer` with `required = true` and `sync-role` is unset. In that case either drop the dependency or set `sync-role: <providerId>` in config.
6. No DB schema changes vs 0.3.3.
