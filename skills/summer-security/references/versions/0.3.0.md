# Summer 0.3.0 — summer-security

Released: 2026-04-15

## Features

### JWT blacklist via Redis

The APISIX resource server can now reject revoked tokens (`401 Unauthorized`) by checking the
JWT's `jti` claim against a Redis key. Enable by setting:

```yaml
f8a.security.apisix.resource-server:
  blacklist-prefix-key: "auth-blacklist"
```

The auth pipeline runs `EXISTS <prefix>:<jti>` on every authenticated request. **Fail-open** on
Redis error — connectivity loss does not block traffic.

Publish a revocation by writing to Redis:

```
SET auth-blacklist:<jti> <any-value> EX <remaining-token-seconds>
```

The next request carrying that `jti` returns 401.

### `JwtBlacklistChecker` strategy interface

Override the default Redis check by providing your own `@Bean JwtBlacklistChecker` (database,
composite, in-memory for tests). `@ConditionalOnMissingBean` skips the default Redis wiring when
a custom bean is present.

```java
@Bean
public JwtBlacklistChecker jwtBlacklistChecker(DatabaseClient db) {
    return jti -> db.sql("SELECT 1 FROM revoked_tokens WHERE jti = :jti")
        .bind("jti", jti)
        .map((row, meta) -> true)
        .first()
        .defaultIfEmpty(false);
}
```

## Breaking changes

### `ApisixAuthenticationManager` constructor signature

The 1-arg and 2-arg constructors are removed. Now requires three args.

```java
// Before (0.2.9)
new ApisixAuthenticationManager(mapper);
new ApisixAuthenticationManager(mapper, converter);

// After (0.3.0)
new ApisixAuthenticationManager(mapper, converter, blacklistChecker);
```

The third arg may be `null` to disable blacklist checking. Spring auto-configuration handles this
transparently — only direct instantiation (tests, custom wiring) needs updating.

## Migration from 0.2.9

1. Tests/configs that instantiate `ApisixAuthenticationManager` directly: pass `null` (or a
   stub) as the third arg.
2. (Optional) enable blacklist by setting `blacklist-prefix-key` and providing a Redis
   connection on the classpath.
3. (Optional) provide a custom `JwtBlacklistChecker` bean if you don't want Redis-backed checks.

## Verification

- `JwtBlacklistChecker` bean present in `ApplicationContext` only when `blacklist-prefix-key` is set.
- Revoking a token (write to Redis) and replaying a request returns 401.
- Logs show `Failed to check blacklist` at WARN with the request still proceeding when Redis is
  down (fail-open).
