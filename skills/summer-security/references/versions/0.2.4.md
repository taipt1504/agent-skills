# Summer 0.2.4 — summer-security

Released: 2026-03-22

## Breaking changes

### Keycloak config moved to a shared block

Connection details moved from `sync-role.*` to a top-level `keycloak.*` block. `sync-role` keeps
only `enabled`. (This was the canonical schema for **0.2.4 → 0.2.9**; replaced by per-provider
schema in 0.3.0.)

```yaml
# Before (0.2.3)
f8a.security.apisix.resource-server:
  sync-role:
    server-url: https://kc.example.com
    realm: master
    client-id: admin-cli
    client-secret: secret

# After (0.2.4)
f8a.security.apisix.resource-server:
  keycloak:
    server-url: https://kc.example.com
    realm: master
    client-id: admin-cli
    client-secret: secret
  sync-role:
    enabled: true
```

### `UserInfoAuthenticationConverter` now returns `Mono`

Return type changed from synchronous to `Mono<UserinfoAuthentication>`. Custom converters must
update their signatures.

```java
// Before
UserinfoAuthentication convert(OidcIdToken token);

// After
Mono<UserinfoAuthentication> convert(OidcIdToken token);
```

`ApisixAuthenticationManager` accepts a `Converter<OidcIdToken, ? extends Mono<? extends UserinfoAuthentication>>`
via constructor. The default `UserInfoAuthenticationConverter` is registered with
`@ConditionalOnMissingBean` — override by providing your own bean.

## Features

### Group-Role Authorization (alternative to `resource_access`)

Resolves roles from group membership instead of the standard JWT `resource_access` claim. Three
cache layers: **L1 Caffeine (per-instance) → L2 Redis Sets (shared, optional) → L3 Keycloak**.

```yaml
f8a.security.apisix.resource-server:
  group-role-authorization:
    enabled: true
    claim-name: role_groups               # JWT claim with group names
    l1:
      ttl: 60s
    l2:                                   # omit entire block = L1-only
      ttl: 5m
      invalidation-channel: "group-role-changes"
      key-prefix: "auth-group-role:"
```

Key classes (added in 0.2.4):
- `GroupRoleResolver` — L1 → L2 → L3 cascade.
- `GroupRoleAuthenticationConverter` — extracts `role_groups` claim, delegates to resolver.
- `GroupRoleFetcher` — `@FunctionalInterface` for L3 source of truth. Default uses Keycloak
  `group-by-path` API. Override by providing your own bean.
- `GroupRoleInvalidator` — Redis pub/sub publisher for cache eviction.

Cache invalidation: publish the group name to the configured channel
(default `group-role-changes`) — listener clears L1 + L2 across all instances.

(Per-provider scoping — `scopeKey` in resolvers and `<scope>:<group>` payload format — comes in
0.3.2.)

### New Keycloak Admin API surfaces

- `keycloak.realm().groupByPath(path)` → `Mono<GroupRepresentation>`.
- `keycloak.groups().get(id).roles()` → `ReactiveRoleMappingResource` with `realmLevel()` and
  `clientLevel(clientUUID)` sub-resources (`listAll()`, `listAvailable()`, `listEffective()`,
  `add()`, `remove()`).
- `keycloak.realm().clientScopes()` — CRUD on realm-level client scopes, plus `protocolMappers()`
  sub-resource.
- `keycloak.clients().get(uuid).getDefaultClientScopes()` /
  `addDefaultClientScope()` / `removeDefaultClientScope()` (and the equivalent optional-scope
  methods).

## Migration from 0.2.3

1. **Restructure YAML**: lift `server-url`, `realm`, `client-id`, `client-secret` out of
   `sync-role:` into a sibling `keycloak:` block. Leave only `enabled: true` under `sync-role:`.
2. **Update custom `UserInfoAuthenticationConverter` (if any)**: return `Mono<UserinfoAuthentication>`
   from `convert(OidcIdToken)`. Tests using `block()` should switch to `StepVerifier`.
3. **(Optional) opt into group-role authorization** by adding the `group-role-authorization` block
   and supplying a `GroupRoleFetcher` bean tailored to your role lookup.

## Verification

- `f8a.security.apisix.resource-server.keycloak.server-url` resolves at startup.
- Logs show `KeycloakRoleSynchronizer` running once at boot when `sync-role.enabled: true`.
- For group-role: `GroupRoleResolver` bean present; pub/sub listener subscribed (only when L2
  Redis is configured).
